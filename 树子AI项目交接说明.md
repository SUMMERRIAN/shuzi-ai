# 树子AI项目交接说明

这份文档用于在新对话、新项目或新 Codex 接手时快速恢复项目上下文。接手者请先阅读本文，再阅读代码，不要重新设计整体架构；后续工作以小步修改、局部优化、持续部署为主。

## 1. 项目定位

树子AI是一个面向学生的个人AI学习教练网站。

核心理念：

> 让AI更懂你，它就可以更好的帮助你。

网站不是单纯聊天工具，而是围绕学生长期学习数据建立个人学习档案，逐步完成：

- 了解学生：通过学情问卷、学情陈述、试卷/错题材料等收集信息。
- 形成画像：AI统一分析学生的学习状态、问题来源和阶段性重点。
- 制定策略：根据学情画像生成科目策略、学习任务和资料使用建议。
- 安排执行：制定学习计划，记录每日反思和每周讨论。
- 错题训练：上传错题/试卷，分析错因，生成类似题，沉淀错题档案。
- 知识笔记：学生提出任意知识问题，AI生成讲解或知识图。
- 长期保存：每个学生有独立数据库档案，后续可不断更新。

## 2. 代码仓库与部署

GitHub 仓库：

```text
https://github.com/SUMMERRIAN/shuzi-ai
```

服务器部署路径：

```bash
/var/www/shuzi-ai
```

服务器更新命令：

```bash
cd /var/www/shuzi-ai
git pull origin main
npm install
npm run build
systemctl restart shuzi-ai-api
systemctl reload nginx
```

本地常用命令：

```bash
npm install
npm run dev
npm run build
node --check server/index.js
```

重要提醒：

- `.env` 不允许提交到 GitHub。
- API Key、数据库密码、JWT Secret、管理员密码只能放服务器环境变量或 `.env`。
- 大型资料文件、PPT、学生原始材料不要放进代码仓库。

## 3. 技术架构

前端：

- React 19
- Vite
- lucide-react 图标
- 主要文件：`src/main.jsx`
- 样式文件：`src/styles.css`
- AI提示词与档案构建：`src/aiAgent.js`

后端：

- Node.js
- Express
- PostgreSQL
- Multer 文件上传
- OpenAI SDK
- Gemini API 通过 HTTP 调用
- 主要文件：`server/index.js`
- 数据库结构：`server/schema.js`
- 上传处理：`server/uploads.js`

部署：

- Nginx 托管前端 `dist`
- Nginx 将 `/api` 转发到 Node 后端
- Node 后端通过 systemd 服务 `shuzi-ai-api` 常驻运行
- PostgreSQL 保存用户、学生档案、会员、Token、上传文件、错题、日历、资料库等数据

## 5. 核心页面逻辑

### 首页

展示项目核心理念、整体学习流程和常见学习问题拆解。

首页偏介绍和引导，不能放太复杂的操作。

### 学情问卷

面向学生填写，采用分步骤问卷。

逻辑：

- 核心必填
- 科目专项按需展开
- 支持保存草稿
- 提交后写入学生档案
- 可作为学情画像分析的数据来源

### 学情陈述

学生主动说明自己的学习问题。

逻辑：

- 可以选择科目、场景、影响程度。
- 可以填写文字陈述。
- 可以用麦克风录音。
- 可以上传已有语音。
- 录音或上传语音后，先保存到个人档案，再尝试转写。
- 同一个音频文件不能重复分析。
- 麦克风录音停止后自动上传、保存、转写。
- 转写结果作为学情陈述记录进入学生档案。
- 学情陈述本身不在本页做完整画像分析，统一交给学情画像页处理。

### 学情画像

这是AI统一分析学生情况的页面。

AI画像的数据来源只允许：

- 学情问卷
- 学情陈述
- 每日反思
- 每周讨论

明确不要把以下内容作为画像证据：

- 错题专项
- 知识笔记
- 学习日历
- 学习资料库
- 学习社区
- AI自由问

输出应包含：

- 阶段性总结
- 核心判断
- 可能原因
- 证据
- 标签
- 追问问题
- 下一步建议
- 维度评分

### 策略与任务

按科目生成学习策略、学习任务和资料使用建议。

逻辑：

- 参考学情画像。
- 每个科目独立。
- AI只处理当前科目的策略、任务、资料使用、完成标准。
- 不要重新做全局学情诊断。
- 不要生成学习计划表。
- 不要处理错题图片。

### 学习计划

包含：

- 每周学习计划表
- 本周重点训练的学习方法
- 本周重点培养的学习习惯
- 每日反思表
- 每周讨论表

逻辑：

- 学生可以确定空闲时间。
- AI可以根据策略与任务给学习计划建议。
- 学生可以自行修改。
- 学习计划、方法习惯培养表可以下载/打印为PDF。
- PDF里不要出现无用的页面说明、按钮、图标、打勾框等界面元素。
- PDF只打印实际启用的学习方法或学习习惯。

### 错题专项

已重做为类似 Gemini/ChatGPT 的工作台。

顶部菜单：

- AI处理
- 错题档案

AI处理区：

- 一个输入框
- 可上传图片、PDF、Word、Excel、PPT、文本等材料
- 可写补充说明
- 快捷按钮：
  - AI分析错题
  - AI生成类似题
  - AI分析试卷

逻辑：

- 错题专项默认使用 Gemini。
- 简单错题分析默认 Gemini Flash。
- 复杂试卷分析可用 Gemini Pro。
- 输出像聊天客户端一样干净、完整、结构化。
- 支持导出 Word 文档。
- 错题和分析结果进入学生错题档案。
- 错题档案支持单选、多选、滚动查看、打包下载PDF。

### 知识笔记

学生提出任意知识问题，AI生成讲解或知识图。

逻辑：

- 可问学习问题，也可以问学习之外的知识问题。
- 知识图生成必须使用 OpenAI 图像模型。
- 目前环境变量使用 `OPENAI_MODEL_IMAGE`，目标模型为 `gpt-image-2`。
- 学生可以查看、修改提示词模板。
- 点击“套用知识图模板”时，将模板与学生输入主题融合。
- 点击“AI生成知识图”后才开始生成。
- 图片预览区右侧使用明显文字按钮“下载图片”。

知识图提示词模板要保持专业教材风格，注意已经把“生物教材风格”改为“教材风格”。

### 学习日历

模仿 Notion 日历。

逻辑：

- 点击日期格可以选中日期。
- 点击“新建页面”在选中日期内创建标题卡片。
- 双击/点击日期格可快速创建。
- 点击标题卡片打开居中的子页面弹窗。
- 子页面可修改标题、日期、正文。
- 可上传 TXT、图片、PDF 等附件。
- 附件可以在子页面下方预览，不新开浏览器页。
- 图片/PDF支持放大查看。

### 学习资料库

模仿 Google Drive。

左侧分区：

- 首页
- 我的云端硬盘
- 最近用过
- 已加星标
- 回收站
- 存储空间

逻辑：

- 支持新建文件夹。
- 支持上传资料和上传文件夹。
- 支持图片、PDF、音频、视频、TXT、Word、Excel、PPT 等。
- 图片/PDF/音频/视频/TXT 可以在页面内预览。
- Word/Excel/PPT 不做复杂在线预览和编辑，保留下载和学习笔记/备注/整理内容。
- 每个文件可绑定学习笔记/备注/整理内容。
- 文件列表需要名称、类型、大小、修改时间、操作对齐。
- 文件名过长要省略，不能挤压其他列。
- 删除按钮必须有效。
- 预览/备注区域只绑定当前选中文件，切换分区后不应继续显示。

### 学习社区

类似学习论坛/贴吧。

逻辑：

- 会员才可以发言、发帖、留言。
- 游客可以浏览。
- 分区包括其他人的帖子、我自己的帖子、向版主提问。
- 主要功能是学习问题分享、学习心得、向版主提问、评论互动。

### AI自由问

独立的自由提问页面。

逻辑：

- 学生可以问学习、作业、知识、生活、兴趣或任意突然想到的问题。
- 可以上传图片或文件。
- 支持选择模型提供商：OpenAI 或 Gemini。
- 支持快速/思考模式。
- 已处理重复提交问题：同一内容短时间内不会反复自动发送。
- 只回答当前问题和当前附件，不进入学情画像。

## 6. 会员与支付系统

当前是手动收款、管理员确认的模式。

用户逻辑：

- 游客可以浏览页面。
- 需要AI分析、下载、保存个人档案、错题训练等能力时，需要登录并开通会员。
- 登录方式已改为用户名/密码为主，不依赖邮箱验证码。

会员套餐：

- 月付
- 季付
- 半年付
- 年付

Token充值：

- 100元
- 300元
- 500元
- 自定义金额

计费原则：

- 页面上不要向学生展示真实API成本、倍率、美元成本等后台成本信息。
- 当前业务设定为约 3.5 倍成本倍率，但这是后台经营逻辑，不展示给用户。

支付流程：

1. 用户选择会员套餐和/或Token充值金额。
2. 系统计算总金额。
3. 用户确认后才显示微信/支付宝收款码。
4. 用户付款后点击“我已充值”。
5. 页面提示：请在夏雨老师群里告知夏雨老师充值完成。
6. 管理员人工确认付款。
7. 管理员在后台为用户开通会员、设置到期时间、增加Token。
8. 只有管理员确认后，用户侧才显示正式充值记录。

会员状态：

- 未登录
- 已登录，未开通会员
- 已开通会员
- 会员到期

会员时间：

- 管理员输入会员开始日期和套餐。
- 系统自动计算到期日期。
- 到期后自动取消会员资格或标记过期。
- 用户侧应显示剩余时间。

管理员中心：

- 普通会员不能使用。
- 左下角有“管理员中心”入口。
- 页面文案只说明“管理员专用入口”，不要告诉普通用户后台可以修改Token等敏感功能。
- 管理员账号/密码通过服务器环境变量配置。
- 不要在前端代码里硬编码管理员密码。

## 7. AI接口与模型分工

### OpenAI

用于：

- 学情画像
- 策略与任务
- 学习计划
- AI自由问中的 OpenAI 模式
- 语音转写
- 知识图生成

关键环境变量：

```env
OPENAI_API_KEY=
OPENAI_MODEL_TEXT=
OPENAI_MODEL_TRANSCRIBE=
OPENAI_MODEL_IMAGE=gpt-image-2
```

### Gemini

用于：

- 错题专项
- AI自由问中的 Gemini 模式

关键环境变量：

```env
GEMINI_API_KEY=
GEMINI_MODEL_FAST=gemini-2.5-flash
GEMINI_MODEL_THINKING=gemini-2.5-pro
```

### 接口边界原则

每个AI按钮只做当前页面的任务：

- 学情画像：只做画像，不生成策略和计划。
- 策略与任务：只做当前科目策略/任务，不做画像，不排课表。
- 学习计划：只做计划建议，不重新诊断。
- 错题专项：只分析当前上传材料和当前任务。
- 知识笔记：只生成当前知识问题的讲解或知识图。
- AI自由问：只回答当前问题，不写入学情画像。

### 防重复提交

已知必须保持：

- AI自由问不能因为状态变化重复发送。
- 语音上传/录音不能重复分析同一个音频。
- 所有AI按钮都应由用户明确点击触发，不要放在 `useEffect` 中自动循环。

## 8. 数据库与学生档案

核心原则：

- 每个学生有独立 `student_id`。
- 所有与学生相关的信息都应绑定用户和学生。
- PostgreSQL 是正式数据存储。

重要表包括：

- `users`
- `students`
- `student_memberships`
- `learning_token_wallets`
- `token_transactions`
- `payment_orders`
- `storage_quotas`
- `uploaded_files`
- `student_archive_events`
- `student_intake_questionnaires`
- `student_statements`
- `statement_audio_files`
- `student_learning_profiles`
- `mistake_files`
- `generated_practice`
- `knowledge_notes`
- `learning_calendar_events`
- `library_items`
- `forum_posts`
- `forum_replies`

学情画像只读取：

- 问卷
- 陈述
- 每日反思
- 每周讨论

其他内容可以保存进档案，但不应自动影响学情画像。

## 9. UI与文案要求

整体风格：

- 简洁
- 美观
- 大气
- 清晰
- 适合学生使用
- 灰、白、黑、蓝、绿为主色系

文案原则：

- 不要讲太多技术或后台成本。
- 不要让学生看到API费用、倍率、美元成本等经营信息。
- 每个页面要让学生知道“这里要做什么”，但不要堆叠无用说明。
- 下载/打印按钮只放在真正需要下载打印的地方。
- 页面顶部大块介绍区如果与内容重复，应删除或简化。
- 操作按钮要明确，不要出现“指向不清楚”的按钮。

排版原则：

- 不要让文字挤压、溢出。
- 长文件名要省略。
- 表格列要对齐。
- 大量内容要放在可滚动容器里。
- 移动端和小屏幕要能看到全部菜单。

## 11. 协作规则

接手 Codex 必须遵守：

- 用户经常要求“先讨论规划，再执行”，如果用户没有明确说开始执行，先不要写代码。
- 用户明确说“开始执行”“请修改”“请推送”时，再进行代码操作。
- 每次修改后必须自检。
- 常规自检包括：

```bash
npm run build
node --check server/index.js
git diff --check
```

- 修改完成后要说明改了什么、验证了什么。
- 除非用户明确要求，否则不要自动推送 GitHub。
- 用户要求推送时，先确认差异范围，再提交并推送。
- 不要回滚用户已有修改。
- 不要把 `.env`、密钥、收款码敏感路径、大型资料文件提交到仓库。
